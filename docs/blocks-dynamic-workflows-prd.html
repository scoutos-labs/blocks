Blocks — skill-shaped no-code blocks for deterministic agent workflows

PRD · task slug blocks-dynamic-workflows · repo /Users/tron/desk-projects/blocks · created 2026-07-02 · status: plan approved (blocker B1 resolved, suggestions S1–S4 applied)

Summary

Blocks is a library of composable, skill-shaped building blocks plus a tiny zero-dependency CLI that lets agents assemble repeatable tasks into a deterministic DAG (a workflow), save it as a file, and run it over and over. Deterministic steps are executed by real code; non-deterministic steps (judging, classifying, extracting, summarizing) are clearly-marked fuzzy blocks whose interface is still deterministic — a prompt contract with a validated output schema. A workflow whose nodes are all fuzzy prompt-contract blocks is a prompt DAG; mixed workflows are the common case.

Core definition (project-specific)

The central design law: the CLI is the runtime for deterministic nodes; the agent is the driver and the oracle for fuzzy nodes. If determinism depended on an agent faithfully following prose, determinism would be a promise, not a property. Here every determinism claim is machine-checkable: run a workflow twice, diff the run-state hashes.

Goals

  • Repeatable agent tasks become saved, re-runnable, auditable workflow files.
  • Determinism enforced by code (argv exec, permission allowlists, hash-recorded run-state), not by prose.
  • Fuzzy reasoning (LLM-as-judge, classify, extract) is first-class but fenced: schema-validated interfaces, gate-bounded blast radius.
  • Blocks are skill-format-compatible: frontmatter uses only documented Claude Code skill keys, and blocks link installs a block into .claude/skills/ via symlink. Everything is plain inspectable files.
  • Zero runtime dependencies beyond Node ≥ 18; the CLI never calls a model.

Non-goals (v1)

  • No hosted server, web UI, daemon, triggers, or schedules (not CI — composes with CI).
  • No loops/recursion/arithmetic in the DAG (node-local bounded retry ≠ a DAG loop); no dynamic fan-out; no parallel scheduler (topo-sequential).
  • No block registry/marketplace; no version ranges (exact pins only).
  • No container sandboxing — allowlists + argv discipline, stated honestly.
  • The CLI never invokes a model; the agent is the only fuzzy oracle (headless --model adapter is v2).

Constraints (security & privacy)

  • Command injection: inputs pass as argv arrays / env-file only; the CLI never builds shell strings. Fixture: binding value "; rm -rf ." must arrive as a literal argument.
  • Path escape: declared path globs rejected at validate time if absolute or containing ..; runtime path values (which may flow from bindings) resolved+normalized and rejected at exec time with a test fixture (../ESCAPED.md, /tmp/ESCAPED.md → exit 3, nothing written).
  • Prompt injection: fuzzy output can only take the declared shape; a poisoned judge at worst picks a wrong declared branch, never runs undeclared code. Documented residual risk.
  • Secrets: no {{env.*}} bindings in v1; run-state never persists process env; runs/ gitignored; grep check in validation suite.
  • Tampered blocks: workflow grants ∩ block permissions; run-state block hashes make drift auditable.
  • Determinism theater: network: false default; double-run hash diff in the validation suite.
