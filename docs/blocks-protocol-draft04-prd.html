Blocks Skill Protocol Draft 04

Turn Draft 03’s sound model and the completed runtime-hardening pass into a reproducible, custody-safe, independently auditable protocol.

Definition

Draft 04 is the trustworthy-ledger draft. A run remains the source of truth, byte-exact evidence is reproducible outside JavaScript, an approval can be signed without handing the runner a private key, secrets are not linkable across runs by an unsalted digest, and paused work is discoverable without parsing console prose.

Draft 04 does not add fan-out, N-of-M approval, a daemon, model invocation, a network sandbox, vendoring namespaces, a general expression language, or remote publication. Those remain experiments or non-goals. Publishing to ZenBin and committing are separate user-approved actions.

Baseline and constraints

  • The worktree contains a completed, reviewed, uncommitted runtime-hardening change set. Draft 04 builds on it; it must not revert it.
  • Pre-existing unrelated files are left unchanged from their inherited Draft-04-start state: context.md, docs/blocks-explainer*.html, and docs/blocks-explainer.html.bak. blocks-protocol.html is a generated protocol artifact and may be regenerated only at the final step.
  • Private keys and raw secrets must never enter run files, examples, progress docs, test output, or the repository.
  • Every new normative MUST has implementation evidence and an adversarial test. No paper-only state-machine design.
  • Protocol-3 runs remain historical artifacts. The Draft-4 runner refuses cross-draft resume/record rather than restamping them.

Plan

1 · Freeze baseline and write Draft-04 red evidence

Cover non-BMP key ordering, 1e21, -0, string escaping, both block-hash preimages, approval payload/signature, gate behavior, protocol gating, detached forgery, same-secret cross-run unlinkability, classifier out-of-set refusal, mixed joins, status, discovery, and writer safety.

  • Acceptance: existing npm run ci remains green before Draft-04 tests are enabled; Draft-04 tests fail only for named missing behavior; unrelated baseline files are unchanged.
  • Evidence: baseline status snapshot, red test transcript, vector inventory.

2 · Reproducible canonicalization and Draft-4 cryptographic preimages

Make canon.js a strict RFC 8785 implementation for JSON-domain values, with UTF-16 code-unit key ordering; rejecting non-finite/unsupported values is an input guard, not an extension of JCS. Centralize approval/secret preimage construction. First prove whether RFC 8785 is byte-identical to the current canonicalizer for all committed Draft-3 approvals. If it is not, retain a draft-scoped legacy path in audit; never reinterpret historical bytes.

  • Acceptance: every independent vector reproduces; Draft-3 approval vectors still verify under Draft-3 rules; Draft-4 payload construction has one implementation path used by record and audit.
  • Evidence: focused vector tests and audit tests.

3 · Detached approval and external key custody

Add a read-only command that emits the exact approval signing payload for a paused node plus --output <candidate.json> and an --approval <file> record path accepting only keyId and signature. Verify before answer-contract accounting. Keep --sign only as a local convenience, but reject any private-key path inside the workspace. Change new key to require or choose a private output outside the workspace. Containment checks use resolved/real paths for both workspace and key (including symlink and .. cases) while still writing only the public registry document to keys/.

  • Acceptance: an external signer can export and sign the payload for the exact candidate answer, then complete a run; forged/wrong-context signatures exit 3 with no attempt burn or mutation; detached record never opens private material; broad block reads cannot reach the private key; no private key is tracked.
  • Evidence: process/file-open spy test, path-fence test, signed run audit.

4 · Close silent-failure and persistence gaps

Add an output-schema relation enumFromInput (protocol-4-only, output-only, references an array input of the same scalar type), amend [SCH-1]/[SCH-8]/[BLK-12], and enforce it in record/audit; update classify. Extend check-output with --input <resolved-input.json>; if a contextual relation exists and input context is absent, return a usage error rather than a false pass. Reject gates mixing join kinds. Persist secretSalt, mark it a [RUN-8] determinism exemption, use the exact domain-separated preimage above, and verify resumed secrets against it. Add atomic write-then-rename and a documented one-writer-per-run rule; do not overclaim that rename alone prevents lost updates.

  • Acceptance: an out-of-label classifier answer cannot be recorded; ordinary enum/schema behavior is unchanged; mixed joins fail with pointer and fix hint; equal secrets in two runs have different stored digests and still resume; interrupted replacement cannot truncate a valid run; concurrent mutation is refused or explicitly outside the conforming caller contract.
  • Evidence: adversarial tests, workflow validation sweep, concurrency probe.

5 · Status, discovery, and normative hardening

Add structured execution results without parsing prose. The derived run status is failed if any direct/descendant node failed; paused if a direct pending fuzzy record has persisted input and blockHash or a descendant is paused; complete if every node is done/skipped and declared outputs resolve; otherwise pending. Add blocks runs [--json] over the workspace run directory; paused records expose the [RNR-8] submission target and required ceremony without raw secret-derived input. Update Draft-04 normative text for all completed hardening behavior.

  • Acceptance: status/discovery are read-only and deterministic; paused descendant runs point to the child run; duplicate run IDs in the workspace inventory are explicit findings and ambiguous entries are not silently merged; malformed/legacy/future runs are reported, not crashed on; every new normative statement maps to a test.

7 · Adversarial review and release-ready gate

Review as a second-language implementer, security reviewer, and large-workflow author. Repair blockers, then rerun the full suite. Do not publish or commit without a separate user request.

  • Acceptance: no required reviewer finding; correctness/security/maintainability pass; taste and originality each score at least 4/5; npm run ci and git diff --check pass; no secrets, private keys, temp files, or unrelated edits appear.
  • Evidence: progress ledger, reviewer report, final status/diff summary.

Migration rules

  • New run documents use protocol: 4. Draft-4-only schema relations and mixed-join refusal require Draft 4.
  • Canonicalization, secret digests, and approval-domain behavior are interpreted by the run’s declared draft. Historical bytes are never rewritten.
  • Detached signing does not itself alter the signed fields. The approval domain tag changes only if the signed preimage changes; the decision must be captured in §20 and vectors.
  • Secret salts never enter fuzzy approval preimages because secret-derived values are forbidden from fuzzy inputs.
  • Run IDs are unique within one workspace run inventory. Generation remains UUID-based; blocks runs detects duplicates and reports them instead of merging ambiguous records.
