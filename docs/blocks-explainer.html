The repeatable parts run as code. The judgment is a contract any agent can sign.

Agents are great at figuring a task out once — and unreliable at doing it the same way the fiftieth time. Blocks saves the task as a DAG of typed blocks that a tiny, zero-dependency CLI executes identically every run, and fences the parts that genuinely need judgment into fuzzy blocks: prompt contracts whose answers are schema-checked before the run will accept them. 1Read the run back

Watch where the work happened. Three nodes — log, render, publish — ran in the CLI lane: exact commands and pure transforms, executed by the blocks CLI itself, each stamped with a hash of the block that ran. Two nodes — draft and judge — are fuzzy: the run paused, an agent read the block's prompt contract and answered, and the answer entered through a schema check. The gate in the middle was decided by the CLI, comparing recorded numbers. The agent never got a vote it didn't sign.

the CLI is the runtime for deterministic nodes;
the agent is the driver, and the oracle for fuzzy nodes.

That law is what makes determinism a property instead of a promise. Every run is a file. 2A block is literally a skill

A block is a directory with two files. SKILL.md is a real Claude Code skill — frontmatter restricted to documented keys, so blocks link can symlink it straight into .claude/skills/. contract.json is the machine-readable truth: typed inputs and outputs, and for deterministic blocks, the exact command and a permissions allowlist. The two inks below are the whole taxonomy.

Fuzzy content is generated; the interface is not. The outputs schema in contract.json is what blocks record enforces — an answer that doesn't fit is named field-by-field and bounced back, three attempts total.

3Five words carry the system

block

One unit of work: a skill-shaped directory, deterministic or fuzzy, with a typed contract.

wire

{{nodes.log.output.text}} — a binding from one node's output to another's input. Wires define the DAG's edges.

gate

A when condition on a node, in a grammar too small to hide logic in: ref op literal, joined by and/or. Draft 3 adds contains and a # length modifier — still no parens, no functions.

run

One execution, persisted as a state file: statuses, outputs, attempts, block hashes. Resumable, diffable, auditable.

grant

The workflow co-signing a block's declared permissions. Effective capability is the intersection — grants never expand. Parents co-sign an embedded workflow's grants too.

Gates cut paths — and a skipped path is a result, not an error. In this repo's other example, a bug-report run classified the report p1, so the backlog lane was skipped by the CLI, with the reason on the record:

4Any agent can be the oracle

There is no SDK to integrate and no adapter to write. When a run reaches a fuzzy node, the CLI pauses and prints the entire protocol — which contract to read, what the input is, and the exact command that continues the run:

Anything that can read a file and run a command can drive a workflow — the protocol doesn't know who you are:

CLAUDE CODE

Ships with two skills in this repo: blocks-composer builds workflows, blocks-runner executes them.

SDK AGENT

An agent loop that shells out: exec, answer the contract, record, repeat until run complete.

ANY VENDOR

No model calls inside the CLI, no API keys, no telemetry. The oracle is whoever answers the contract well.

A HUMAN

Read the SKILL.md, type JSON in an editor, run blocks record. Slow oracle; same guarantees.

And because answers are schema-checked at the door, a weak oracle degrades loudly, not silently: an invalid answer is bounced with the exact violating field, three attempts total, then the node fails and the run stops.

When a node demands more than an answer — a signing key with declared claims, an attested capability — the pause banner prints the complete ceremony, --sign and --attest included, exact strings filled in. The protocol never makes the oracle guess what authority looks like. That story is FIG. 7.

5Compose, then demand a key

The protocol's next two drafts answered two questions. Can workflows nest? Once a workflow declares typed outputs, it has the same interface shape as a block — so it can be a node. The entire FIG. 1 pipeline is now one framed node inside the release workflow below, and the parent must co-sign everything the child may touch: the root workflow's grants are a complete statement of what the whole tree can do. Can approval be authoritative? A fuzzy block can demand that its answer arrive signed by a registered key carrying declared claims, and attested to a declared capability. The runner refuses anything less at the door — without burning the attempt budget.

6Guarantees, honestly stated

  • Bindings are filled as whole argv elements — never through a shell. A value of "; rm -rf ." arrives as those literal bytes. There's a test.
  • Effective capability = block declarations ∩ workflow grants; ungranted binaries are refused, paths are workspace-fenced at validation and again at exec, and entry scripts run under Node's permission model where available.
  • network: false is a reviewed declaration, not a sandbox — v1 has no containers, and says so.
  • Fuzzy content is never trusted: schemas bound its shape, gates bound its blast radius, and prompt contracts tell the oracle to treat inputs as data, not instructions.
  • Approvals are Ed25519 signatures over a domain-tagged string binding workflow, block, run, node, input, and answer. Record-time verification is enforced; whether a key's claims are true is a reviewed registry statement — declared, and the sheet says so. Capability attestations follow the same honesty: equality enforced, truth declared.
  • Hashes are draft-scoped: a run is read under the protocol draft it declares. Older committed runs are history, not drift.
  • Non-goals, kept on purpose: no loops or fan-out in the DAG, no parallel scheduler, no registry, no server, no PKI. The CLI never calls a model.
