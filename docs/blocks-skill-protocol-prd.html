Blocks Skill Protocol — full draft spec (PROTOCOL.md + zenbin)

PRD · slug blocks-skill-protocol · 2026-07-02 · deliverables: PROTOCOL.md (repo root, Draft 01) + published draft at zenbin · status: plan approved

Definition

An interoperability-grade specification of the Blocks skill protocol: a third party could implement conforming blocks, workflows, runners, oracles, or composers from this document alone, without reading this repo's code — and they would interoperate. It formalizes SPEC.md (which stays normative for this repo's internals) and resolves every prose/code discrepancy explicitly. RFC-2119 keywords; stable requirement IDs ([BLK-n]…); every MUST machine- or transcript-checkable. Draft 01 — no stability promise; strict unknown-key rejection is the extension policy.

Key editorial decisions (fixed)

  • D1 precedence: PROTOCOL.md normative for interop behavior; SPEC.md normative for repo internals; conflicts are bugs, tracked in Appendix D. SPEC.md gets a one-line scoping cross-reference.
  • D2 pause interface: information items normative (node id, block pin, SKILL.md location, resolved input — persisted in the run document as the copy of record — and how to submit); exact printout text informative (Appendix A).
  • D3: verb names informative; the four outcome classes normative, and mapped to exit codes 0/1/2/3 for any CLI-shaped runner.
  • D4: strict JSON, camelCase run keys, kebab names/ids; unknown keys rejected everywhere — strictness is the compatibility mechanism. Run serialization (2-space, insertion order, trailing LF) RECOMMENDED.
  • D6 determinism: structural equality of parsed .nodes for det nodes is MUST; byte identity RECOMMENDED.
  • D7: examples are verbatim repo artifacts, marked informative, byte-checked by the harness so the spec cannot rot from its own examples. The protocol must not require JavaScript — a Python runner must be able to conform.
  • Conformance classes (6): document classes Block / Workflow / Run; actor classes Runner / Oracle / Composer. No Validator class (validate-only tools satisfy the Runner's static-validation subset); no registry.
