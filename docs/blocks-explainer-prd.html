Blocks explainer — animated, self-contained HTML

PRD · task slug blocks-explainer · created 2026-07-02 · deliverable docs/blocks-explainer.html + published copy on zenbin.org · status: plan approved

Definition

A single self-contained animated HTML document that explains the design and idea behind Blocks and shows that any agent (Claude Code, SDK agents, other vendors, a human) can drive it, because the whole interface is files + CLI verbs. It is an engineering document, not a marketing site: every number, hash, and output on the page comes from a committed artifact in this repo. The page must render from file://, with JS disabled (final-state fallback), behind a strict CSP (inline everything), and honor prefers-reduced-motion.

Design direction (fixed)

  • Concept: one drafting sheet — engineering-drawing title block (doc name, REV, date, SHEET 1 OF 1) instead of a hero banner; figures numbered like plates (FIG. 1 — RUN r-269b010f).
  • Palette: paper #F2F5F7, blue-black ink; det = drafting blue, fuzzy = amber-bronze applied as a consistent two-ink system everywhere (schematic, code coloring, vocabulary strip); pass-green for results. No cream/terracotta, no dark+acid-green, no gradients.
  • Type inversion: ui-monospace wherever the machine speaks (headlines, labels, CLI output, stamps); humanist system stack for narration only.
  • Signature (the one animated element): the real changelog-from-git DAG executing as a stepped schematic: det nodes stamp sequentially with real 8-char hash chips, run breathes amber at fuzzy nodes until an answer clicks through the schema gate via a visible blocks record command, the when gate is drawn as a comparator evaluating 0.85 >= 0.7 in the CLI lane. Autoplay on first visibility + prev/next controls; pauses offscreen. Everything else: quiet scroll reveals only.
  • Copy voice: plain, specific, active; five-word vocabulary exactly as the repo uses it; README honesty register kept near-verbatim.

Plan — 6 steps

Step 1 — Skeleton + design system

  • AC: opens from file:// with JS disabled; exactly one h1; monotonic heading levels; zero external refs (<link>, <script src>, http(s) in src/href/url()/@import); all 6 sections with final copy; schematic present as static final-state SVG.
  • Evidence: node self-containment check; browser open.

Step 2 — Real-artifact content pass

  • AC: every embedded snippet byte-matches its repo source (modulo trim/HTML-escape) and carries an HTML comment naming the source path; no invented hashes/scores/outputs; five words used only in the repo's sense.
  • Evidence: scratchpad node script diffs snippets against sources; grep for any hash/score absent from examples/runs/*.json.

Step 3 — The orchestrated schematic

  • AC: ~8-state stepped machine; state order matches the runner protocol exactly (no parallel det stamps; fuzzy answers arrive via visible record; gate evaluated in the CLI lane); reduced-motion renders completed final state; JS-disabled default = completed run; no timers while offscreen; keyboard-operable step controls.
  • Evidence: node check for prefers-reduced-motion + IntersectionObserver and no unconditional setInterval; reviewer steps through in browser.

Step 4 — Quiet reveals + accessibility

  • AC: body ink on paper ≥ 7:1; det/fuzzy ink text usages ≥ 4.5:1; reveals gated behind reduced-motion; semantic landmarks; visible focus.
  • Evidence: node contrast computation on declared tokens; keyboard tab-through.

Step 5 — Validation harness + budget

  • AC: one scratchpad node script asserts self-containment, size ≤ 200 KB (target 150), one h1 + sane headings, reduced-motion present, title+meta description present, no secrets (sk-, AKIA, ghp_, Bearer ) and no /Users/ paths.
  • Evidence: all-pass checklist output.

Step 6 — Publish to zenbin.org

  • AC: API discovered at publish time from zenbin.org; harness re-run on exact upload bytes; published URL renders identical to local (or graceful static fallback if CSP blocks inline JS).
  • Evidence: published URL; local-vs-published comparison.

Taste / originality criteria (score ≥ 4 required)

  • Fails if: gradient hero, emoji icons, drop-shadow card grids, fake macOS terminal with traffic lights, marketing verbs, a checkmark where a comparator belongs, any number not traceable to a run file.
  • Passes if: reads as one drafting sheet (title block, FIG plates); the two-ink det/fuzzy split works as a system a reader can decode anywhere on the page; hash chips read as inspection stamps; type inversion is felt; exactly one animated element; CLI output rendered as a ruled spec plate, not skeuomorphic terminal.

Constraints / security

  • Self-contained: no external requests of any kind; publishable behind strict CSP.
  • No secrets, no absolute /Users/ paths, no emails in the published page.
  • The animation must not lie: no parallel det stamps, no oracle answer without record, no agent-judged gates.
  • Not marketing: no CTAs/testimonials/benchmarks; only run-file numbers.
  • SPEC.md stays normative; the doc says so.
